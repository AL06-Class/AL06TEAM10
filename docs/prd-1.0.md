# PRD 1.0: 피트니스 트레이너 검증 채용 플랫폼

## 1. Summary

이 문서는 피트니스 센터 대표/관리자가 검증된 트레이너를 빠르게 찾고 비교할 수 있는 채용 플랫폼의 1.0 제품 요구사항입니다.

첫 버전은 채용 실패 리스크를 줄이는 데 집중합니다. 센터 대표가 트레이너의 프로필, 케이스 테스트 결과, 평가 데이터를 보고 면접 전 역량을 판단할 수 있게 합니다.

트레이너(`candidate`) 역할의 온보딩, 케이스 테스트, 인증, 채용 제안 대응 흐름은 `docs/prd-trainer-flow-1.0.md`를 하위 PRD로 따릅니다.

## 3. Background

### 문제

피트니스 센터 대표는 트레이너 채용이 급한 상황에서 빠르게 실력 있는 인재를 뽑으려 합니다. 하지만 이력서와 자격증만으로는 회원 지도 역량, 수업 스타일, 매출 기여 가능성을 판단하기 어렵습니다.

잘못 뽑으면 3개월 정도의 시간 낭비, 재교육 비용, 회원 이탈, 운영 손실이 생깁니다. 평균 채용 실패 1회당 손실은 수백만 원대로 가정합니다.

### 기회

피트니스 업계에는 트레이너 역량을 객관적으로 비교할 수 있는 신뢰 기준이 부족합니다. 이 플랫폼은 검증된 트레이너 프로필, 평가 문항, 케이스 테스트 결과, 포트폴리오 템플릿을 묶어 채용 전 판단 근거를 제공합니다.

### 팀 강점

- 전직 피트니스 관리자 경험
- 트레이너 전문성을 데이터로 시각화하고 태그화하려는 방향
- 센터의 재등록률, 유지율 같은 현장 데이터를 성과 예측에 활용하려는 방향

## 4. Objective

### 목표

직원 5~15명 규모 피트니스 센터의 대표/관리자가 채용 실패 리스크를 줄이고, 면접 전에 트레이너 역량을 비교할 수 있게 합니다.

### 왜 중요한가

센터 대표는 채용을 빠르게 해야 하지만, 실력 검증 없이 사람을 뽑으면 교육 비용과 회원 이탈 위험을 떠안습니다. 이 제품은 채용 전에 판단 근거를 제공해 시간과 비용 손실을 줄입니다.

### Key Results 초안

- 첫 버전에서 센터 대표가 추천 트레이너 5명을 한 화면에서 비교할 수 있다.
- 첫 버전에서 각 트레이너의 프로필, 인증 여부, 케이스 테스트 결과, 추천 이유를 확인할 수 있다.
- 첫 버전에서 센터 대표가 후보를 고른 뒤 채용 제안을 보내는 흐름까지 이동할 수 있다.
- 검증 가설: 채용 성사 1건당 30~50만 원을 지불할 의사가 있는지 확인한다.

## 5. Market Segment(s)

### 1차 고객

- 직원 5~15명 규모 피트니스 센터의 대표/관리자
- 트레이너 퇴사, 신규 오픈, 회원 증가 등으로 빠른 채용이 필요한 사람
- 이력서와 자격증만으로는 실력을 판단하기 어렵다고 느끼는 사람

### 고객이 하려는 일

채용 실패 없이 우리 센터에서 매출을 낼 수 있는 검증된 트레이너를 빠르게 찾는 것입니다.

### 제약

- 센터 대표를 플랫폼으로 끌어오는 초기 네트워크가 필요합니다.
- 트레이너가 자기 데이터와 검증 자료를 실제로 입력하게 만들어야 합니다.
- 트레이너 실력, 수업 스타일, 티칭 퀄리티를 평가할 객관 기준이 필요합니다.
- 첫 버전에서는 커뮤니티, 게시판, 메시징, 채팅은 제외하고 이메일/전화로 대체합니다.

## 6. Value Proposition(s)

### 핵심 가치

피트니스 센터 대표의 채용 실패 리스크를 검증된 트레이너 평가·인증 시스템으로 줄입니다.

### 고객이 얻는 것

- 검증된 트레이너 인재풀 접근
- 객관적 역량 평가 결과
- 후보별 진단 결과 비교
- 채용 성공 확률 향상
- 채용 및 교육 비용 절감

### 고객이 피하는 것

- 이력서와 자격증만 보고 뽑는 불확실성
- 면접 후에도 실력을 알기 어려운 상황
- 채용 실패로 인한 재교육 비용
- 회원 이탈과 매출 손실

## 7. Solution

### 7.1 UX / 첫 사용 흐름

1. 센터 대표가 채용 조건을 설정합니다.
2. 플랫폼이 맞춤 트레이너 5명을 추립니다.
3. 센터 대표가 각 후보의 프로필과 케이스 테스트 결과를 비교합니다.
4. 면접 전에 실력을 파악한 뒤 채용 제안을 보냅니다.

### 7.2 Key Features

#### 검증 인력 프로필

- 사진 및 경력
- 활동 지역
- 인증 트레이너 여부
- 자격증
- 전문 분야
- 성과 데이터
- 케이스 테스트 결과
- 트레이너를 추천하는 이유
- AI 한줄 요약

#### 평가 문항과 포트폴리오 템플릿

- 케이스 테스트 응시 화면
- 테스트 남은 시간
- 테스트 회원 정보
- 테스트 문항
- 제출하기 버튼
- AI 채점 결과 안내

#### 역량 기반 트레이너 검색·필터

- 전문 분야, 경력, 지역에 맞는 트레이너 탐색
- 후보별 진단 결과 비교
- 검증된 트레이너 풀에서 선택

#### 채용 제안 흐름

- 고용 형태
- 급여 조건
- 근무 시작일
- 제안 메시지
- 수수료 안내

### 7.3 Technology

- 앱 구조: React, TypeScript, Vite 기반 SPA
- 스타일: Tailwind CSS
- 역할 화면: 지원자, 채용담당자, 면접관 역할 기반 분기 유지
- Firebase: 지금은 Hosting 중심으로 사용하고, 추후 Firestore, Storage, Functions는 필요 시 추가
- Firebase 설정값: `.env`에만 저장하고 PRD에는 적지 않음
- 환경변수 공유: `.env.example`에는 변수 이름만 기록

### 7.4 Data 기준

- 역할 값은 `candidate`, `recruiter`, `interviewer`를 사용합니다.
- 새 컬렉션, 필드, 상태값, 역할 값은 구현 전에 `docs/data-guide.md`의 공통 이름 사전에 먼저 추가합니다.
- 필드명은 영어 `camelCase`를 사용합니다.
- 상태값은 자유 텍스트가 아니라 정해진 값만 사용합니다.
- 화면 컴포넌트와 데이터 접근 코드는 분리합니다.

### 7.5 Design 기준

- 참고 레퍼런스: Apple 웹 디자인 참고안
- 분위기: 깔끔하고 미니멀한 프리미엄 업무 도구 UI
- 합의 한 줄: Apple식 절제감 + 액션 블루 + 검증된 트레이너 프로필/평가 데이터 중심
- 메인 색상: 액션 블루 계열 `#0066cc`
- 배경: 흰색, 옅은 회색 `#f5f5f7`
- 텍스트: 차콜 계열 `#1d1d1f`
- 과한 장식, 장식용 그라데이션, 과한 그림자는 사용하지 않습니다.
- 프로필 사진, 인증 자료, 평가 결과처럼 신뢰를 만드는 실제 정보를 선명하게 보여 줍니다.

### 7.6 Assumptions

- 센터 대표는 검증된 트레이너 인재풀에 돈을 낼 의사가 있습니다.
- 트레이너는 자신의 검증 데이터와 포트폴리오를 입력할 동기가 있습니다.
- 케이스 테스트와 평가 문항은 실제 채용 판단에 도움이 됩니다.
- AI 한줄 요약과 추천 이유는 센터 대표의 비교 시간을 줄여 줍니다.
- 초기 버전에서는 이메일/전화가 커뮤니케이션을 대체할 수 있습니다.

## 8. Release

### Version 1.0 범위

- 센터 대표/관리자 중심 흐름
- 트레이너 추천 5명 비교
- 검증 인력 프로필
- 케이스 테스트 결과 확인
- 평가 문항과 포트폴리오 템플릿
- 역량 기반 검색·필터
- 채용 제안 보내기 흐름 진입
- Firebase Hosting 배포 기준 유지

### 후속 버전 후보

- 커뮤니티, 게시판, 메시징, 채팅
- 실제 결제와 수수료 정산
- Firestore 기반 실데이터 저장
- Storage 기반 포트폴리오/인증 자료 업로드
- Functions 기반 자동 평가 또는 알림
- 센터 성과 데이터 기반 추천 고도화
- 화상 면접 시스템
- 채용 공고 작성
- 센터에서 원하는 역량을 트레이너 능력치에 가중치를 두어 알고리즘 추천
- 트레이너 케이스 테스트 임시저장
- FAIL 후 재응시 쿨다운/문항 풀 운영 정책

### 협업 운영 기준

- 팀장: 최영
- AI 작업은 개인별 작업 브랜치에서 진행합니다.
- 브랜치 이름은 `student/이름/작업명` 형식을 사용합니다.
- PR은 필수이며, AI가 제목, 설명, 변경 요약, 데이터 사전 변경 여부, 확인 명령, 남은 이슈를 작성합니다.
- 팀장은 AI 리뷰 결과를 참고해 최종 승인하고 `main`에 병합합니다.
- 기능 브랜치는 PR 단위로 하나씩 병합합니다.
- 작업 단위는 가능한 한 독립 구현하고, 데이터 이름 사전은 공유합니다.
- 디자인의 약한 불일치는 후속 정리로 맞출 수 있지만, 공통 구조를 깨는 변경은 후속으로 미루지 않습니다.
