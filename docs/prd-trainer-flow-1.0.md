# PRD 1.0: 트레이너 플로우

상위 문서: `docs/prd-1.0.md`

이 문서는 트레이너(`candidate`)가 가입 후 프로필을 만들고, 케이스 테스트로 실무 역량을 인증받고, 채용 제안을 확인해 수락 또는 거절하는 공급자 측 유저 플로우 기준입니다.

## 1. Summary

첫 버전은 트레이너가 SNS 활동량이나 인맥이 아니라, 자기 프로필과 케이스 테스트 결과로 실무 역량을 증명하는 흐름에 집중합니다.

트레이너 플로우의 목표는 세 가지입니다.

- 온보딩에서 경력, 활동 지역, 자격증, 전문 분야, 성과 데이터를 낮은 마찰로 입력하게 합니다.
- 케이스 테스트를 통해 평가 능력, 운동 처방, 커뮤니케이션, 안전 고려 역량을 점수와 피드백으로 보여 줍니다.
- 인증 결과를 마이페이지와 채용 제안 흐름에 연결해 트레이너가 채용 협상 근거로 사용할 수 있게 합니다.

## 2. User Goal

트레이너는 내 실력을 객관적인 근거로 증명해서 원하는 조건으로 센터와 협상하고 싶습니다.

## 3. Scope

### In Scope

- 트레이너 온보딩 3단계
- 케이스 테스트 안내, 응시, 제출
- 채점 결과 PASS/FAIL 확인
- 인증 완료 화면
- 인증 전/후 마이페이지
- 받은 채용 제안 목록과 상세
- 제안 수락/거절
- 채용 확정 후 담당자 연락처 안내

### Out of Scope

- 커뮤니티, 게시판, 실시간 메시징, 채팅
- 실제 결제와 수수료 정산
- Firestore 실데이터 저장
- Storage 기반 인증 자료 업로드
- Functions 기반 실제 AI 채점

첫 버전의 연락 흐름은 상위 PRD 기준에 맞춰 이메일/전화 안내로 대체합니다. UI에 "대표에게 메시지 보내기" 버튼을 넣지 않습니다.

## 4. Main Flow

1. 트레이너가 `candidate` 역할로 진입합니다.
2. 기본 정보, 전문 분야, 성과 데이터를 입력합니다.
3. 온보딩 완료 후 케이스 테스트를 바로 응시하거나 나중에 하기를 선택합니다.
4. 케이스 테스트 안내에서 소요시간, 채점 영역, 인증 기준을 확인합니다.
5. 테스트 응시 화면에서 회원 케이스를 보고 서술형 답변을 제출합니다.
6. 채점 결과가 80점 이상이면 인증 통과, 80점 미만이면 인증 미달로 처리합니다.
7. 인증 통과 후 마이페이지에 인증 상태와 점수가 표시됩니다.
8. 트레이너는 받은 채용 제안을 확인하고 수락 또는 거절합니다.
9. 수락하면 채용 확정 화면에서 담당자 연락처와 다음 안내를 확인합니다.

## 5. Screen Requirements

### 5.1 온보딩 1/3: 기본 정보

- 필수 입력: 이름, 활동 지역, 경력
- 선택 입력: 주요 자격증, 한 줄 소개
- CTA: 다음
- 검증: 필수 입력값이 없으면 다음 단계로 이동하지 않습니다.

### 5.2 온보딩 2/3: 전문 분야

- 전문 분야는 1개 이상 선택해야 합니다.
- 각 전문 분야는 `proficiencyLevel` 1~5 척도를 사용합니다.
- 선택하지 않은 분야는 `미선택` 상태로 보여 줍니다.
- 특화 케이스는 선택 입력입니다.

### 5.3 온보딩 3/3: 성과 데이터

- 입력 항목: 누적 지도 회원 수, 평균 재등록률, 평균 PT 지속 기간
- 성과 데이터는 선택 입력입니다.
- CTA: 케이스 테스트 바로 응시하기, 나중에 하기
- 나중에 하기를 선택하면 인증 전 마이페이지로 이동합니다.

### 5.4 케이스 테스트 안내

- 안내 내용: 소요시간, 케이스 1개 제시, 서술형 답변, AI 채점, 인증 기준
- 채점 영역: 평가 능력, 운동 처방, 커뮤니케이션, 안전 고려
- 임시저장이 없을 경우 시작 전 명확히 알립니다.
- CTA: 테스트 시작하기, 나중에 하기

### 5.5 케이스 테스트 응시

- 상단에 남은 시간을 표시합니다.
- 회원 정보와 답변 영역을 분리합니다.
- 제출 전 확인 모달을 띄웁니다.
- 브라우저 이탈 또는 뒤로가기 시 작성 내용이 사라질 수 있음을 경고합니다.

### 5.6 채점 결과

- 공통 표시: 종합 점수, 영역별 점수, AI 피드백, 인증 기준 대비 결과
- PASS: 인증 완료 화면으로 이동하는 CTA를 제공합니다.
- FAIL: 재응시하기, 마이페이지로 이동 CTA를 제공합니다.
- 재응시 정책이 정해지기 전까지는 문구를 "재응시 가능 여부는 운영 정책에 따라 안내됩니다"로 둡니다.

### 5.7 인증 완료

- 인증 뱃지, 테스트 점수, 주요 전문 분야를 보여 줍니다.
- CTA: 채용 제안 확인하기, 내 프로필로 이동
- "나에게 맞는 센터 찾기"는 1.0에서 실제 화면이 없으면 표시하지 않습니다.

### 5.8 마이페이지

- 인증 상태, 프로필, 전문 분야, 성과 데이터, 케이스 테스트 이력, 받은 채용 제안을 보여 줍니다.
- 인증 전 empty state를 제공합니다.
  - 제목: 아직 인증 전입니다.
  - 설명: 케이스 테스트를 완료하면 인증 상태와 점수가 프로필에 표시됩니다.
  - CTA: 케이스 테스트 응시하기
- 받은 제안이 0건이면 empty state를 제공합니다.
  - 제목: 아직 받은 제안이 없습니다.
  - 설명: 인증과 프로필 완성도가 높을수록 센터 대표에게 더 잘 노출됩니다.

### 5.9 채용 제안 목록

- 제안 카드에는 센터명, 고용 형태, 급여 조건, 제안일, 상태를 보여 줍니다.
- 상태는 `pending`, `accepted`, `declined`, `cancelled` 중 하나를 사용합니다.
- 제안이 0건이면 empty state를 보여 줍니다.

### 5.10 제안 상세

- 표시 항목: 센터명, 고용 형태, 급여 조건, 근무 시작일, 제안 메시지
- CTA: 수락하기, 거절하기
- 거절 시 확인 모달을 띄우고 상태를 `declined`로 처리합니다.
- 수락 시 상태를 `accepted`로 처리하고 채용 확정 화면으로 이동합니다.

### 5.11 채용 확정

- 센터명, 고용 형태, 시작일, 담당자 이름, 전화번호, 이메일을 보여 줍니다.
- CTA는 이메일/전화 기반으로 둡니다.
  - 담당자 이메일 확인
  - 홈으로 돌아가기
- 실시간 메시징 CTA는 1.0 범위에서 제외합니다.

## 6. Data Contract

데이터 이름은 `docs/data-guide.md`의 공통 이름 사전을 따릅니다.

### Main Collections

- `candidateProfiles`
- `caseTestSessions`
- `hiringOffers`

### Key Status Values

- 케이스 테스트: `inProgress`, `submitted`, `graded`
- 인증 상태: `unverified`, `passed`, `failed`
- 채용 제안: `pending`, `accepted`, `declined`, `cancelled`

채용 제안 거절은 `rejected`가 아니라 기존 상태값과 맞춘 `declined`를 사용합니다.

## 7. Wireframe Revision Notes

전달된 ASCII 와이어프레임과 lofi HTML은 구조 초안으로 유지하되, 구현 전 아래를 보정합니다.

- primary 버튼은 실제 구현에서 검정이 아니라 디자인 시스템의 액션 블루 `#0066cc`를 사용합니다.
- 숙련도 표기는 1~5 척도와 텍스트 수준이 일치해야 합니다.
- 인증 전 마이페이지, 제안 0건, FAIL 후 재응시 정책 안내를 추가합니다.
- 테스트 제출 전 확인 모달과 이탈 경고를 추가합니다.
- "대표에게 메시지 보내기"는 이메일/전화 연락 안내로 교체합니다.
- "나에게 맞는 센터 찾기"는 실제 1.0 화면이 없으면 인증 완료 화면에서 제거합니다.

## 8. Open Questions

- 케이스 테스트 임시저장을 1.0에 포함할지
- FAIL 후 즉시 재응시를 허용할지, 쿨다운을 둘지
- 동일 케이스 반복 노출을 허용할지
- 성과 데이터 자기신고값을 후속 버전에서 어떻게 검증할지
- 채점 영역에 리스크 관리/기대치 조율을 별도 항목으로 분리할지
